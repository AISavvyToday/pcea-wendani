### 1. Core user model, roles, and auth flows (finish + wire to UI)

Goal: **Every login lands on the correct dashboard and only sees what they should.**

We’ll treat this as the next immediate implementation step.

#### 1.1. Authentication flows

- **Login/logout (email + password)** using custom `User`:
  - Implement working `login_page`, `logout_view`, `role_redirect` (we’ve already sketched this).
  - Ensure `LOGIN_URL`, `LOGIN_REDIRECT_URL`, `LOGOUT_REDIRECT_URL` in settings.
  - Ensure login template uses theme but posts `email` + `password` to the view.

- **Password reset flow (later in this step)**
  - Use Django’s built‑in password reset views with custom templates styled via the Purple theme.
  - Configure email backend (for now local console backend; later real SMTP).

#### 1.2. Roles & permissions

- Define clear behavior per `UserRole`:
  - `SUPER_ADMIN`, `SCHOOL_ADMIN`: full access dashboards + config.
  - `ACCOUNTANT`: finance, payments, invoicing, reports subset.
  - `TEACHER`: academics (attendance, marks, class lists) + some reports.
  - `PARENT`: children’s info, balances, statements, results, notifications.
  - `STUDENT`: (optional) personal timetable, results, notices.

- Implementation:
  - `login_required` on all portal views.
  - Simple role‑based decorators or helper (e.g. `role_required([...])`) for sensitive views.
  - Context processor to expose `request.user.role` and role flags in templates, so we can conditionally show menu items.

#### 1.3. Navigation and UX consistency

- Build the **real left sidebar** menu based on role:
  - Admin: full menu (Students, Academics, Finance, Payments, Comms, Reports, Settings).
  - Bursar: Finance, Payments, limited Reports.
  - Teacher: Students (class list), Academics, limited Reports.
  - Parent/Student: minimal portal (profile, balances, statements, results, announcements).

- Reuse theme partials from your HTML files (buttons, dropdowns, tables) to ensure consistent Purple styling.

✅ **Outcome of Step 1:**
You can create users with roles in Django admin, log in through `/auth/login/`, be redirected by role, and navigate only to allowed sections. Once this is stable, we build features under those sections.

### 2. Student Management module

Goal: **Have a working “Students” area where admins/teachers can manage students, guardians, and current enrolment.**

This is the foundation; everything else (academics, finance, comms) hangs off “student”.

#### 2.1. Core stories

- Admin can:
  - View a paginated/filterable list of students.
  - Create a new student (with basic bio + admission number + class + house).
  - Attach parents/guardians and mark which is primary.
  - View a student profile page (with tabs: profile, academics, finance, communications).
  - Promote students from one class/year to the next.

- Teacher can:
  - View their own class(es) list (from `Class`/`StudentClassEnrollment`).
  - Open a student profile in read‑only mode (or limited edit: e.g. remarks).

#### 2.2. Backend implementation

- **URLs & views** (`students/urls.py`, `students/views.py`):
  - `students:list` – list, search, filter by class/stream/status.
  - `students:create` – student registration form.
  - `students:update/<id>` – edit.
  - `students:detail/<id>` – profile view.
  - `students:promote` – simple view/service to promote selected students.

- **Forms**:
  - `StudentForm` for main student info.
  - `ParentInlineForm` or separate view for linking parents (or use dedicated `StudentParentForm`).

- **Services** (optional but clean):
  - `StudentService.create_student_with_parents(...)`.
  - `StudentService.promote_students(...)`.

#### 2.3. UI using Purple theme

- List views:
  - Base them on `basic-table.html` from the theme.
  - Add search bar, filter dropdowns (class, year, status), and actions column.

- Forms:
  - Use `basic_elements.html` / `buttons.html` for styling.
  - Possibly use a **multi‑step form** for more complex data (Bio → Contact → Parent/Guardian → Transport/Medical).

- Student profile:
  - Use a “profile + tabs” layout:
    - Overview tab: basic info, class, status.
    - Academics tab (later pulls marks, attendance summary).
    - Finance tab (later pulls invoice/balance summary).
    - Communications tab (messages sent to this student/parent).

✅ **Outcome of Step 2:**
School admin and teachers can manage students from the web UI. Data model becomes “alive” and ready for linking with invoices, attendance, and results.

### 4. Finance/Fee Management module (with payments tied in)

Goal: **Turn your finance models + payment integration into a working fee/invoicing and receipts module.**

This is where your payment code becomes visible to users.

#### 4.1. Core stories

- Admin/Bursar can:
  - Configure fee structures per class/year/term (`FeeStructure`, `FeeItem` etc. per your models).
  - Bulk generate invoices for all active students for a term.
  - View invoices and balances per student.
  - See a statement of account for a student.
  - See bank transactions (Equity/Co‑op) and how they linked to students/invoices.
  - See any **unmatched bank transactions** or anomalies.

- Parent can:
  - See child balances and statements (read‑only).
  - View breakdown of fees.

#### 4.2. Implementation units

- **Configuration**:
  - Portal UI for `FeeStructure`:
    - Per class/year/term.
    - Use forms reusing theme components.

- **Invoicing service & UI**:
  - Service: `InvoicingService.generate_invoices_for_term(term)`:
    - Reads students in current term, `FeeStructure`, discounts.
    - Creates `Invoice` + `InvoiceItem`s.
  - UI action in bursar dashboard:
    - “Generate invoices for Term X” with confirmation and preview.

- **Invoices & statements UI**:
  - Student finance overview page:
    - Current balance, prepayment, balance B/F.
    - Table of invoices & payments (past X terms).
  - Invoice list view:
    - Filter by class, term, status (paid, partial, overdue).
  - Statement view:
    - A chronological ledger of invoices and payments.

- **Bank transactions & reconciliation**:
  - Views for `BankTransaction`:
    - All transactions; filter by date, gateway, status.
    - Show whether each transaction has a `Payment` / `Invoice` link.
  - View for **unmatched** / **unreconciled** transactions:
    - Let bursar manually assign a transaction to a student/invoice or mark as error.

✅ **Outcome of Step 4:**
Finance/bursar can fully manage fees and see the impact of bank payments without touching the Django admin. Parents can see what they owe.

  - `SECURE_SSL_REDIRECT`, `SESSION_COOKIE_SECURE`, `CSRF_COOKIE_SECURE` on production.

#### 7.3. Tests & CI/CD

- Add unit tests for:
  - Payment flows (Equity/Co‑op callbacks -> BankTransaction -> Payment -> Invoice updates).
  - Fee generation and balances.
  - Admissions / promotions.
- Set up:
  - GitHub Actions (or similar) to run tests on push.
  - Optional: automatic deploy to Heroku on main branch with passing tests.
